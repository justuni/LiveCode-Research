Introduction to RevIPC

Short overview

In this document I’ve tried to compile my thoughts in reaction to the items discussed regarding Inter-Program Communication so far. I’m not even going to try and pretend it’s a fair nor a complete overview. I’m just trying to make sure we’re all on the same level.

Hence your input is vital. I think I’ve got it right, but I may be misinterpreting certain ideas myself. So fire away if something is either unclear or I messed up somewhere.

RevIPC libraries : a hierarchy

There are many aspects to inter-program communication. There are different approaches, depending on the goals: platform-specific, network-oriented, intranet versus internet,...

To take these different goals into account, we’ll have to define a hierarchy of functions that should allow us to tackle each of the sub-problems one at  time, without blocking the use if the parts that have been converted intio a usable format.

Table 1 describes a proposed hierarchy, whereby Revolution Inter-program Communcation is written in a way that allows it to use different ways of communicating between different programs. It should issue standardised calls to pre-registered libraries, so that the developer can decide which features to choose from which vendor in order to program his application.

Bridges

One of the goals of this group should be to provide the ‘bridges’ between RevIPC, which should be protocol-independent, and the different protocols that are available in either a platform-specific or a platform-independent manner. We must also provide ‘hooks’ so that others can add new protocols, and provide authentication and encryption features that are either protocol-specific or work transparently.

Not only does this allow us to integrate better with existing applications, but it could also open up a market for ‘bridge’ applications that will connect a Windows-based application with a MacOS-only application that understands AppleEvents – much like Lasso did to allow web-specific applications to connect to FileMaker Pro databases long before that got its Web Companion.

Protocols

Platform-specific protocols :

-         MacOS : communicating directly via AppleEvents shouldn’t prove too hard, especially if we build our own class ‘RIPC’ and have separate ID’s for messages, remote procedure calls, etc.

-         Windows : Dynamic Data Exchange (DDE) is deprecated, and someone should check whether or not we can use the DDE-externals to setup a ‘listening’ part for DDE or not ; or does anyone have special ideas for .COM externals?

-         Un*x : Process I/O is something I haven’t done before, and it sounds like there’s really no ‘standard’ way of packaging your data and requests but it would prove ideal in tying our applications with existing programs on these platforms.

Platform-independent protocols :

-         File-based : for some purposes, it could be easier to simple write data to a file in a shared directory and read that file into the other application ; advantage is that we’re really not depending on the presence of network-protocols.

-         TCP-based : so far most of our attention has gone to TCP-based communcation, and STAMP was defined from this perspective ; ideal in an internet-environment but not necessarily the most optimised.

-         UDP-based : for applications that are aimed at communicating over a local network and don’t have to rely on the sequence of the requests, UDP may be a simpler option than TCP.

Perspective : server-side

This part of the document is written from the perspective of an application that is offering services to other programs. In fact, it will cheerfully ignore the client-side of the service process.

Any references to that side are for informational purposes only and not definitive in any way shape or form. A separate document will be supplied shortly with regards to the client-side in this process.

Service points

Definition

A service point is a part of an application that interfaces with other programs through a common protocol, offering services to those other programs that may range from simple event-messaging to complex remote procedure calls.

One could compare it with a city hall, where you can come and ask for questions, fill in forms, and start certain procedures. In small cities, there’s one counter and it handles everything.

In larger cities, there will be more than one counter, and these are usually specialised: one handles births, deaths and marriages ; another handles building permits ; etc.

In such cases, there’s usually a desk where you can ask what type of service is offered where. And each counter will have flyers at hand describing its services, and forms to fill in order to use these services.

Main service point

By default, a “main service point” will be setup. This main service point can be asked what other service points there are and where ; and as it is a service point by definition, it can also be asked what services it offers itself.

Additional service points

You don’t have to deploy any additional service points, as one service point may be all you need. But whereas most applications will run fine with a single service point, there may be reasons to split up its services and divide them over different service points:

-         Security reasons, as some services are offered only to certain people, and it’s safer to have a security officer at each counter than at the front desk.

-         To make things more manageable by forming a front office for bundled services that act as a whole.

-         Some services are handled differently than others. Some require rigorous FIFO scheduling, others are aimed at handling all requests as fast as they can, regardless of when they were received.

Nevertheless, all these service points can interact when the need arises, much like the city hall has an internal telephone system. Moreover, they can use common resources, much like the city hall may have a centralised library or a mail service.

Sidebar :: TCP-based protocol, service points and ports

For more than one reason, it is easiest to say that one service point takes one port on the IP-address of the computer the application is running on. The main service point will always be in a certain range, to make discovery an acceptable task.

Other service points will have ports in a different range. As stated before, the main service point will be able to tell on what ports the other service points can be talked to.

Service points and objects

The easiest way to bundle services, is by storing them into one object, and connecting this object to the service point, and thus to one port. This also allows us to more easily handle dispatching the requests.

Another reason for this, is that it makes it easier to determine what services are offered on a point: a plug-in could be built to look at the script of an object, present the developer with a list of handlers and functions, letting him or her decide which are internal and adding parameter information to others.

This service point information can easily be stored in the custom properties of the object itself, and handed out to potential service users in a standardised format. As a bonus, it makes it easier for the dispatcher to see what is and what is not a legitimate remote procedure call for that service point.

Blocking and Scheduling

To block or not to block

For some purposes, such as record updating in a database server, it is essential that the requests are processed sequentially, and that no one can access the data during this period in time. Other activity is put on hold until this request is completed, and no new requests can be started.

Other services however, may be less picky about how the requests are processed, and will execute them as they arrive. Or other services may use a system of priorities, where all ‘urgent’ chores must be handled before less urgent chores. However, this does not stop other requests from coming in.

Approaches to scheduling

In an early post [message number 35], I came to a list of five approaches. This list is not exhaustive, but it may be all we need for our purposes.

  1. Process asap, regardless of order and origin
  2. Process sequentially if from the same origin
  3. Process sequentially ; accepting new calls
  4. Process sequentially ; blocking new calls
  5. Process according to priorities

Regardless of which approach is appropriate, there must be a low-level dispatcher capable of handling these different approaches, in order to accomodate different needs.

Scheduling per service point

As there are many different approaches to scheduling, there is probably also a need for mixing these approaches in one and the same application, over multiple service points.

One may build an application where some services give priority to requests from the intranet, and other services just try to treat every request as soon as possible, regardless of origin.

And even then, it may want a set of services to temporarily take over and process something quickly and urgently, halting the other services during this process.

Sidebar :: file-based protocol and scheduling

A file-based protocol would regularly check a shared directory for the presence of any new files. In a priority-driven schedule, it would first check for files of a certain type, and handle these first ; it could go through the files in alphabetical order or sort the information on loast modification date, etc.

Authentication and Encryption

Authentication

A topic which hasn’t been discussed in much depth, is authentication. So far, we’ve briefly considered:

-         LDAP

-         RADIUS

-         Proprietary authentication scheme

Whichever way we go, at some point we’re going to have to pick one or more of these. It would be best if a few of us form a subcommittee to take the time to look into this matter more throughly, and describe pros and cons of each.

Perhaps it’s best to shield authentication off into its own layer, so we can substitute one method for another, depending on the application. However, in view of the hierarchy proposed at the start of this document, it is very well possible that a developer could have to mix and match different approaches.

For example, an AppleEvent scheme could require no authentication features at all, as the MacOS has its own simple method of accepting and rejecting remote connections, by allowing users to connect to programs or not. In case of a file-based protocol, the administrator can set user rights.

Encryption

A topic on which we’ve spent quite a few posts, but haven’t really decided on which way to go, especially as each approach has its pros and cons.

-         SSL [would be great, but depends on when RunRev gets this ability, and then there’s still the question of its server-side abilities]

-         PGP [would be good, but who will implement this cryptography gem?]

-         Proprietary encryption scheme [would be less of a hassle, and immediately available, but could be cracked]

Perhaps it’s best to shield encryption off into its own layer, so we can substitute one method for another, depending on the application. However, in view of the hierarchy proposed at the start of this document, it is very well possible that a developer could have to mix and match different approaches.

For example, in a file-based protocol you wouldn’t use SSL but you could be interested in PGP, and for an intranet application a proprietary scheme could be enough whereas the internet version would need SSL.
