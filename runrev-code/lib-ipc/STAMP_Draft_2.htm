Introduction to STAMP

STAMP – the acronym

STAMP stands for Simple Transfer, Authentication and Messaging Protocol.

Simple
  • Easy to use
  • Easy to configure

Transfer
  • Text data
  • Binary data

Authentication
  • Authentication
  • Encryption

Messaging
  • XML-envelope
  • Targeted

STAMP – the goal

STAMP strives to enhance the communication between programs, facilitating the transfer of text and binary data, enabling remote procedure calls wrapped in an XML-envelope, incorporating authentication and encryption features.

STAMP – the protocol

STAMP in its current form, borrows heavily from the FTP and SOAP protocols.

Functional

Quick overview

STAMP is a proposal architecture for communication between applications. A connection is opened, authentication processed, and then the connection is used to send back and forth STAMP-streams during one or more sessions.

Multiple layers

A library will provide the high-level access, hiding away the complexity involved in sending the streams back and forth, so that the developer needn’t be aware of the potential pitfalls of sending back and forth:

  • Text data that may have been polluted with tags or reserved characters and may disrupt the XML-parsing process.
  • Binary data that might disrupt the XML-parsing process as well.
  • XML-encoded remote procedure calls, and the serialisation process required to make it work.

Besides the simple library handlers, we’ll need a series of internal handlers and functions that will

  • Translate the high-level calls into series of low-level calls
  • Provide callback handlers for the communication handling
  • Take care of the wrapping if the data and the remote procedure calls

Low-Level

The STAMP-stream structure

All STAMP-streams start with a three digit code [StreamType], followed by additional parameters on the same line.

This first line can be used to determine the type of the message and what data is sent after that first line, as well as its size (always / in certain cases ?).

Some StreamTypes have no additional data, others are used for XML-encoded messages, others are used to denominate the transfer of binary or text data.

The StreamType code

We separate the StreamType codes in two halves:

100 through 499 are client-side StreamTypes

500 through 899 are server-side StreamTypes

The same application can be both a client and a server in this protocol.

In this way we avoid having to set up and maintain two separate socket-connections.

Reserved StreamTypes

100 through 199 / 500 through 599

are reserved for connection-related messages

200 through 249 / 600 through 649

are reserved for authentication-related messages

250 through 299 / 650 through 699

are reserved for encryption-related messages

300 through 349 / 700 through 749

are reserved for transfer of text data

350 through 399 / 750 through 799

are reserved for transfer of binary data

400 through 499 / 800 through 899

are reserved for transfer of XML data

 

Why the separate codes for text, binary and XML data?

In order to make it easier for the STAMP-dispatcher to parse the data sent through and decide how to process it.

Moreover, in order to fully support RPC, we need a mechanism that will:

  1. Send the RPC, packaged in XML
  2. Send some text and all binary data separately, as they might interfere with the XML-structure

STAMP-dispatcher

The STAMP-dispatcher reads the first line, and then leaves the rest of the handling to the correct library callback-handler.

These library callback-handlers should send messages (STAMPEvent ?) to the appropriate place in the message chain and/or call handlers and function directly in the case of RPC’s.
